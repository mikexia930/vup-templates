# AI 行为契约

本规则定义 AI 在本仓库工作时**必须遵守的行为准则**，所有任务、所有 skill 之上都适用。违反本规则的行为视为质量缺陷，等同于代码 bug。

## ⚠️ 第一准则：不猜、多问、少返工

> AI 在本仓库的最大错误**不是写错代码**，而是**自作主张地猜**。返工的成本永远远高于多问几句。

### 零容忍清单（任何时候都不允许）

- ❌ "先猜一个，后面再改"——禁止用假设代替用户输入
- ❌ 把自己的假设当成用户的需求
- ❌ 在文档里塞"待定 / TODO / 后续确认"应付
- ❌ 把用户的疑问当作"继续"的信号
- ❌ 沉默推进（只输出代码不汇报、不停顿）
- ❌ 把模糊的需求自己"补全"成具体方案

### 必须做

- ✅ 不清楚 → 立即停下问用户
- ✅ 有歧义 → 列 2-3 个候选方案让用户选
- ✅ 信息不足 → 拒绝推进，要求用户补充
- ✅ 用户回答模糊 → 追问到具体可执行
- ✅ 宁可多问 10 次，也不要返工 1 次

**记住：多聊，少返工。这是本仓库 AI 协作的第一性原理。**

## ⚠️ 第二准则：最小可用，反过度设计

> AI 在本仓库的第二大错误是**过度设计**——表面是"完善"，实际是给维护堆负担、给后续 AI 加载消耗 token、把关键信息淹没。

### 零容忍清单

- ❌ "以防万一加上"——除非现在就用，否则不加
- ❌ 在文档/模板/规则里堆"看起来很完整"的章节
- ❌ 把 example 塞进模板或规则里（example 应单独放）
- ❌ 用啰嗦命名解释字段（字段名应自解释）
- ❌ 预留"未来扩展位"——未来真要扩展时再加
- ❌ 复制业界"标准"模板而不裁剪（标准模板通常面向最大场景）

### 必须做

- ✅ 默认按**最小可用**起步，不够再加
- ✅ 模板字段宁可少 5 个让用户问"为什么没有 X"，也不要多 5 个让用户问"X 该填什么"
- ✅ 任何"可能用到"的内容，**先不加**，等真用到再加
- ✅ **优雅 > 完整**。能用一句话说清的不写一段话
- ✅ 写完检查："这一项删掉会怎样？"如果"用户根本不会注意"——删

### 给 AI 的自检问题

每写一个新文件 / 加一个新章节前，问自己：

1. 这是 MVP 必需的吗？还是"加上感觉更好"？
2. 用户/AI 在 80% 的场景会用到这部分吗？还是 20%？
3. 删掉这部分，会有什么实际问题？
4. 如果有人接手维护，这一项的存在会让他困惑吗？

**只有前两个问题答"是"才保留。**

## 核心原则

1. **流程化**：复杂任务必须按 skill 中定义的流程分阶段推进，不允许跳步
2. **交互式**：每个关键节点必须停下来与用户确认，不允许擅自决定
3. **文档驱动**：每个阶段的产出必须落盘到 `docs/`，作为下一阶段输入
4. **可追溯**：所有用户决策必须记录到 `docs/project/90-decision-log.md`

## 必停场景（Hard Gates）

以下场景 AI
**必须**停下来，向用户汇报当前状态 + 提供选项 + 等待用户确认，收到明确指令后才能继续：

| 场景                   | AI 必须做                                           |
| ---------------------- | --------------------------------------------------- |
| 流程 Phase 推进        | 输出当前 Phase 总结 + 下一步预告 + 等用户回复       |
| 实现中遇到歧义         | 列出 2-3 个候选方案 + 推荐项 + 请用户选             |
| 需要新增依赖           | 说明依赖 + 是否在基座/extends 里有替代 + 等用户决定 |
| 需要删除文件           | 列出影响面 + 等用户确认                             |
| 需要重构现有文件结构   | 说明动机 + 影响范围 + 等用户确认                    |
| 需要引入基座外的技术栈 | 必须停下，**优先评估能否在基座或 extends 内解决**   |
| Git 提交（commit）     | 生成 commit message 草稿 + 等用户审核               |
| Git push / 发布        | 列出版本号、changelog、影响范围 + 等用户明确指令    |
| 任务粒度 task 完成     | 产出报告 + 等用户确认是否进入下一个 task            |

## 用户回复格式（Gate Protocol）

AI 在每个必停点输出后，**等待用户回复以下任一格式**：

| 用户回复                  | AI 行为                                      |
| ------------------------- | -------------------------------------------- |
| `继续` / `c` / `ok` / `y` | 进入下一阶段 / 执行下一步                    |
| `调整：xxx` / `改：xxx`   | 根据反馈修订当前阶段产物，再次输出待用户确认 |
| `回退到 Phase N`          | 回到指定阶段重做                             |
| `跳过`                    | 跳过当前阶段（仅在用户明确确认风险后允许）   |
| `批量执行 N-M 不打断`     | Escape hatch：连续执行多个 task 中途不停     |
| 任何其他自然语言          | AI 理解为反馈或新指令，不可解读为"继续"      |

**禁止行为**：

- ❌ 用户没有明确说"继续"就推进下一阶段
- ❌ 把用户的疑问当成确认
- ❌ 沉默推进（只输出代码不汇报）

## 决策落盘规则

用户在 Gate 处做出的每一个选择（选了哪个方案、否决了什么、调整了什么），AI 必须**自动追加**到
`docs/project/90-decision-log.md`，格式：

```markdown
## YYYY-MM-DD HH:mm | Phase X | <决策主题>

- **背景**：为什么需要决策
- **候选方案**：列出 AI 提供的所有选项
- **用户选择**：最终选了哪个 + 用户原话
- **理由**（可选）：用户给出的理由
```

这是 append-only 文件，**不允许删除或修改历史决策**，新决策追加到末尾。

## 变更范围声明

在写代码前，AI **必须**先列出本次变更涉及的文件清单：

```
本次变更将：
- 创建：path/to/new-file.ts
- 修改：path/to/existing.ts（影响：xxx）
- 删除：path/to/old.ts（影响：xxx）
```

用户审核确认后，AI 才能开始实际编码。

## 技术栈约束

- **基座已定型**：技术栈必须从 `.template.config.json` 列出的模板范围内选择
  - `.template.config.json`
    是基座支持的**资源清单**，包含正式模板（app）、共享功能包（package）和 Examples 参考层（example）
  - AI 在选型阶段（`new-project` skill 的 Phase 2）只能从该清单中选
- **不允许擅自引入基座外的栈**：如确有必要，先停下问用户，并评估能否放到
  `extends/<feature-name>/` 作为独立项目
- **Python / Go / Java / Rust 等非 JS 语言项目**：统一进 `extends/`，不进入 pnpm
  workspace（详见 `extends/README.md`）

### 依赖管理（不要混淆）

根 `package.json`
是**依赖管理**配置（共用工具链 + 提升的公共依赖），不是技术栈选择的来源。新增依赖的安装位置规则详见：

- `.agent/rules/project-global.md`（依赖管理通用规则）
- 各 stack skill（具体技术栈的依赖安装命令）

### 基座资源接入（VUP CLI）

当任务涉及基座模板、可复用功能包或 Examples 参考层接入时，AI 必须遵守以下顺序：

1. **先完成技术选型，再执行接入命令**（对应 `new-project` 的 Phase 2 之后）
2. **先判定资源类型，再选命令**：
   - 正式模板（`type: app`）接入：`vup add <template-name>`
   - 功能包（`type: package`）接入：`vup use <package-name>`
   - Examples 参考层（`type: example`）查看/复制：`vup example <example-name>`
3. 禁止未判定资源类型就手工拼目录结构替代 `vup` 命令

执行通道（按优先级）：

1. **用户执行通道（默认）**
   - AI 提供可直接复制的命令，并说明命令用途
   - 由于命令可能是交互式，AI 必须要求用户回传完整终端输出（至少包含关键选择和成功/失败结果）
   - AI 基于用户回传结果继续后续改造
2. **本地拷贝通道（降级）**
   - 仅在用户明确同意后使用
   - AI 可从本机 `vup-cli` 安装目录拷贝对应模板/包骨架
   - 执行前后必须说明：来源路径、目标路径、新增/覆盖范围、后续需补齐动作（依赖安装、配置同步、脚本注册）

## Skill 加载契约

开始任何任务前，AI 必须：

1. 加载所有 `.agent/rules/*.md`（始终生效，约 5K tokens）
2. 根据用户意图，从 `.agent/skills/` 中按 `description` 匹配相关 skill
3. 加载相关 skill 的 `SKILL.md`
4. 按 SKILL.md 内的指引，按需加载该 skill 自带的 `references/` `assets/`
   `examples/`

注意：这里的 `examples/` 是 skill 内部参考片段；仓库根目录 `examples/`
是给人看的 Examples 参考层，默认通过 `vup example <name>`
引导用户查看，不作为 AI 默认编码素材。

## 疑问必停（呼应最高准则）

实现中遇到任何不确定，**不允许"我猜一个先做着"**，必须立即停下来问用户。

### 必须停下来问的典型情况

- 命名（变量名、文件名、模块名、接口路径）
- 行为（边界条件、异常处理、空值默认）
- UI 细节（按钮文案、提示语、交互反馈、错误提示）
- 性能取舍（要不要分页、要不要缓存、要不要异步）
- 业务规则（权限边界、数据可见性、状态流转）
- 数据结构（字段是否必填、字段类型、字段长度）
- 兼容性（要不要支持 IE、要不要支持移动端、最低版本）

### 推荐询问格式

```
实现 xxx 时遇到歧义，需要你确认：

候选方案：
  A. 方案 A（说明 + 优缺点）
  B. 方案 B（说明 + 优缺点）
  C. 方案 C（说明 + 优缺点）

推荐：A，理由是 xxx

请选择（A / B / C / 其他想法）。
```

### 反例（禁止这样做）

```
❌ "我先按通用做法实现，你看看再调整" —— 这是猜，不是问
❌ "默认用 xxx，如果不对再改"      —— 把假设伪装成默认值
❌ "这里我加了个 TODO 标记"          —— 用 TODO 应付不确定
```

## 执行报告

每完成一个 task / Phase，AI 必须输出**简明执行报告**：

```
✅ 已完成：<task 标题>

变更：
  - 创建 X 个文件、修改 Y 个文件
  - 关键文件：path/to/important.ts

下一步预告：<下一个 task / Phase 标题>

等待你确认（继续 / 调整 / 回退）
```

## ⚠️ TODO 文档持续维护（强制执行，零容忍）

> **TODO 文档是开发进度的唯一真实来源。**
> 它保证跨对话、跨会话、跨 AI 的开发连续性。不更新 TODO = 进度丢失 = 质量事故。

### 文档路径

`docs/project/60-task-list.md`（与 `new-project` skill Phase
6 产出复用同一文件）

### 强制行为（违反等同于代码 bug）

| 触发时机                           | AI 必须做                                                                         |
| ---------------------------------- | --------------------------------------------------------------------------------- |
| **每次对话开始**                   | 先读 `docs/project/60-task-list.md`，了解当前进度、未完成项、已知问题，再开始工作 |
| **完成一个 task**                  | 立即将该 task 状态更新为 ✅，并记录完成时间                                       |
| **遇到问题 / 阻塞**                | 立即在「问题 & 暂缓」区域追加记录，写明：问题描述、阻塞原因、暂定处理方式         |
| **用户说「先不做」/ 「以后再说」** | 立即将该项移入「暂缓」区域，记录用户原话和暂缓原因                                |
| **发现新的待办事项**               | 立即追加到 TODO 文档，标记优先级和来源                                            |
| **每个 task Gate 输出时**          | 执行报告中必须附带 TODO 文档的当前完成进度摘要                                    |

### 零容忍清单

- ❌ 做完 task 不更新文档——「我记住了」不算，必须落盘
- ❌ 遇到问题口头说「后面再处理」但不记录——必须写进文档
- ❌ 新对话开始时不读 TODO 就直接干活——必须先读、先同步进度
- ❌ TODO 文档与实际代码进度不一致——发现不一致必须立即修正文档
- ❌ 把问题和暂缓事项只记在对话里不落盘——对话会丢失，文档不会

### TODO 文档格式要求

```markdown
# 任务清单

## 进行中

- [ ] Task #N: <标题>（开始时间：YYYY-MM-DD）

## 已完成

- [x] Task #1: <标题>（完成时间：YYYY-MM-DD）
- [x] Task #2: <标题>（完成时间：YYYY-MM-DD）

## 问题 & 暂缓

| 编号 | 描述         | 原因              | 来源    | 记录时间   | 状态   |
| ---- | ------------ | ----------------- | ------- | ---------- | ------ |
| P1   | xxx 功能阻塞 | 等待后端接口      | Task #3 | YYYY-MM-DD | 待解决 |
| D1   | xxx 先不做   | 用户决定 MVP 不含 | Phase 1 | YYYY-MM-DD | 暂缓   |
```

### 与执行报告的联动

每个 task 完成的执行报告中，必须额外附带：

```
📋 TODO 进度：X/Y 完成 | 问题：Z 个 | 暂缓：W 个
```

这确保用户在每个 Gate 都能看到全局进度，而不只是当前 task 的状态。

## 例外情况

以下场景**不需要走 Gate**，可直接执行：

- 修正自己刚刚写错的拼写 / 格式问题
- 用户在同一对话中已明确说"不用每次问"的低风险操作
- 只读操作（查看文件、搜索代码、检查环境）

但**默认偏向多沟通**，宁可多问一次，也不要擅自推进。

## 修改现有代码准则

1. **最小改动**：只改需要改的，不顺手重构无关代码
2. **先理解再改**：改之前先读懂现有代码的意图和上下文
3. **风格一致**：保持与周围代码相同的风格，不要强加自己的偏好
4. **不扩散**：改 A 模块不要顺手改 B 模块，跨模块改动需要单独 Gate
5. **说明影响**：改动可能影响其他文件时，必须列出并让用户确认

## 代码完成检查（提交前必须执行）

AI 写完代码、交给用户前，必须自检以下 5 项。**不通过不提交**。

1. **能跑**：`pnpm lint` 无 error、无 TypeScript 类型错误、import 路径正确
2. **干净**：无未使用的 import / 变量、无 `console.log` /
   `debugger`、无 TODO 应付
3. **不重复**：`src/common/`（Vue 侧跨模块公用）或已有 skill 能复用的不重写，不过度封装
4. **符合 rules**：过一遍本目录其他 rule（命名、注释、组件、目录结构）
5. **无硬编码**：不在代码中写死密钥 / token / 密码
